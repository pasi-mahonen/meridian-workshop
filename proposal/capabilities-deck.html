Reports

Filters unwired

Time Period, Warehouse, Category, and Order Status filters are not correctly driving API calls in the Reports view.

Architecture

Migration incomplete

Data layer

Scope of Work

Four requirements.
One team. Eight weeks.

R1 — Required

Reports Remediation

  • Full audit against all ~8 logged defects — document before fixing
  • Rewire all four filters to correctly drive API query parameters
  • Close i18n gaps so the Reports view is fully translatable
  • Migrate Options API to Composition API — consistent with the rest of the frontend
  • Defects beyond the eight become a scope conversation, not silent additions
R2 — Required

Restocking View

  • New /restocking route — consistent with existing navigation and design
  • Operator inputs a budget ceiling; we do the rest
  • Backend combines stock levels + demand forecast → prioritized order list
  • Each row: item, warehouse, current stock, forecast demand, qty, estimated cost
  • Sortable table with total cost indicator — built for daily operational use
R3 — Required

Browser Testing

  • Playwright end-to-end tests — tooling already in the project
  • QA Engineer starts Week 1, not Week 7 — coverage grows alongside the code
  • Critical flows defined at kickoff with Meridian IT — not decided unilaterally
  • Baseline: dashboard load, filter interactions, Reports page, Restocking view
  • Tests committed to repo and runnable in CI
R4 — Required

Architecture Docs

  • Interactive HTML diagram — shareable, no special tools needed
  • Covers: components, data flow, API surface, filter system, deployment
  • Explicitly documents the JSON file data layer and scaling implications
  • Written for Meridian IT — not for us to file away
  • What the previous vendor's two-page handoff was not
